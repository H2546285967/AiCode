## 二、我的真实角色（修正版 2026-06-07）

**不是架构师，不是从零搭建者，而是：在已有架构上做应用层开发维护的Java工程师。**

### 具体工作内容

| 工作内容 | 具体说明 |
|:---------|:---------|
| SSE流式输出对接 | 大模型API返回SSE流 → 逐chunk解析 → 推送给前端，处理过流式和下游节点的冲突 |
| 新增工作流节点 | 在Camunda BPMN引擎上新增自定义节点（如文本提取节点），内部调用大模型 |
| 修改现有节点功能 | 根据需求修改已有节点逻辑 |
| 节点间变量赋值 | 通过Camunda流程变量做节点间数据传递（变量复制、变量提取） |
| 对接算法接口 | 按接口文档调用NL2SQL/RAG等算法接口，组装参数、处理返回 |
| 知识库系统维护 | 维护LakeSearch知识库系统的数据权限、知识库CRUD，RAG底层原理未深入 |
| 跨团队Debug | 出问题时：提取CURL报文 → Postman复现 → 定位是应用侧还是算法侧 |
| 对话流维护 | 维护会话管理、消息存储、多轮对话等模块 |

### 技术栈详情

| 技术 | 具体使用 |
|:-----|:---------|
| 流式输出 | SSE（Server-Sent Events） |
| 工作流引擎 | Camunda BPMN |
| 数据传递 | 流程变量赋值（key-value） |
| 模型调用 | 按算法团队接口文档调用 |
| 问题定位 | CURL + Postman，跨团队协作排查 |

### 我接触过的节点类型

```
变量复制节点 ← 把一个变量的值赋给另一个
变量提取节点 ← 从复杂结果中提取特定字段赋给新变量
文件解析节点 ← 解析上传的文件内容
文本提取节点 ← 调用大模型提取文本信息（新增）
LLM节点      ← 调用大模型生成回答
分类器节点   ← 意图分类/路由
HTTP请求节点 ← 调用外部API
代码沙箱节点 ← 执行用户代码（GraalVM）
回复/输出节点 ← 返回结果给用户
```

## 四、面试能讲的故事（诚实版）

### 故事1：SSE流式输出

```
场景：大模型回答需要实时推送给用户

我做的：
- 对接大模型API的SSE流式返回
- 逐chunk解析，通过SSE推送给前端
- 前端逐步渲染，用户看到"打字机"效果

遇到的问题：
- 流式输出和工作流的回复节点冲突
- 流还没结束，下游回复节点就开始执行
- 导致返回给用户的内容不完整

我的处理：
- 在流式完成前阻塞下游节点执行
- 用完成标志位控制节点间数据流转
- 确保流式内容完整后再交给下游

这个经验让我理解了：SSE流式在实际工程中不只是简单的推送，
还涉及并发控制、状态管理、异常中断处理等问题。
```

### 故事2：新增工作流节点（文本提取节点）

```
场景：需要在工作流中增加文本提取功能

我做的：
- 基于Camunda JavaDelegate接口实现节点逻辑
- 内部组装Prompt模板，调用大模型API
- 从大模型返回结果中提取关键信息
- 通过流程变量把结果传递给下游节点

技术要点：
- 继承Camunda的JavaDelegate接口
- 通过流程变量（Process Variables）做节点间数据传递
- 处理大模型调用的异常和超时
- 输出格式要和下游节点的输入格式对齐
```

### 故事3：跨团队对接NL2SQL

```
场景：智能问数模块对接算法团队的NL2SQL接口

我做的：
- 按算法团队提供的接口文档实现调用逻辑
- 组装用户输入为请求参数
- 拿到SQL结果后执行查询，返回给用户

Debug模式：
- 出问题时从日志提取CURL请求报文
- 用Postman直接调用算法接口复现
- 对比应用侧发出的参数和接口文档要求
- 定位是参数组装问题还是算法模型问题
- 把CURL报文发给算法团队协助排查

这个模式让我快速定位了多个跨系统问题。
```
