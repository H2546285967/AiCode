## 一、新项目投喂建立记忆

### 步骤1：通义灵码对话框拖入四个项目核心文档

**推荐拖入文档类型**：架构设计文档、核心功能需求文档、智能体开发规范、接口说明文档（确保核心性，避免无关文档混入）

```
你是资深Java后端工程师，现在请你精读以上仓颉智能体平台核心文档，系统梳理并记忆以下维度的关键信息：
请先输出一份「仓颉智能体平台核心信息总结」，确认你已完整掌握；
同时将这些核心信息写入项目永久记忆，确保后续新对话中无需重复检索文档，可直接基于记忆回答相关问题。
```

### 步骤2：验证投喂成功（1行指令）

发送以下指令，确认记忆已绑定新项目：

```
请按「智能体配置、技术栈、工作流、RAG检索、LLM调用、发布权限」6大维度，
列出当前新项目中已保存的「仓颉智能体平台」核心记忆要点，
每个维度简要说明核心内容（50字以内）。
```

**验证通过标志**：AI 能按指定维度返回核心内容，每个维度描述简洁且贴合文档实际，无遗漏关键维度：

| 验证维度   | 通过标准                           |
| :--------- | :--------------------------------- |
| 智能体配置 | 包含配置格式、必填字段、示例       |
| 技术栈     | 包含后端框架、数据库、中间件版本   |
| 工作流     | 包含节点类型、流转规则、异常处理   |
| RAG检索    | 包含检索逻辑、向量库类型、参数配置 |
| LLM调用    | 包含模型类型、超时机制、重试策略   |
| 发布权限   | 包含权限等级、审批流程、环境限制   |

## 二、快捷触发记忆（3种极简方式）

### 方式1：关键词快捷触发（无需长指令）

提问时只需带 **1个核心关键词 + 项目名**，AI 自动匹配记忆：

| 场景       | 示例指令                  | 适用场景                    |
| :--------- | :------------------------ | :-------------------------- |
| 工作流相关 | 「仓颉智能体 工作流节点」 | 询问工作流设计/节点配置规则 |
| RAG相关    | 「仓颉 RAG检索流程」      | 询问RAG检索逻辑/参数配置    |
| 编码相关   | 「仓颉 app_type编码」     | 询问编码规则/枚举值含义     |
| SSE推送    | 「仓颉 SSE流式推送」      | 询问SSE实现/异常处理        |
| 技术栈查询 | 「仓颉 技术栈包含哪些」   | 快速核对项目技术栈/版本信息 |
| 发布权限   | 「仓颉 发布权限规则」     | 询问发布流程/审批要求       |
| LLM调用    | 「仓颉 LLM超时配置」      | 询问模型调用参数/重试机制   |

### 方式2：固定快捷指令（复制即用）

把以下指令保存为记事本 / IDEA 笔记，每次提问前粘贴 + 补充问题：

```
【调用仓颉项目记忆】：请基于当前新项目已保存的永久记忆，回答：{}
```

**使用示例**：

```
【调用仓颉项目记忆】：请基于当前新项目已保存的永久记忆，回答：SSE流式推送需要注意什么？
```

```
【调用仓颉项目记忆】：请基于当前新项目已保存的永久记忆，回答：RAG检索的向量库配置参数有哪些？
```

### 方式3：一键刷新记忆（怕遗忘时用）

若担心记忆未触发，先发 1 行指令刷新，再提问题：

```
请优先调用当前新项目中「仓颉智能体平台」的永久记忆，接下来回答我的问题。
```

接着直接提问题（如「技术栈包含哪些？」「LLM调用的超时机制是什么？」），**无需额外说明**。

## 三、日常交互使用—极简协作铁律

### 1、完整版（最稳，适合协作开始或任务切换时发送）

```
请严格遵守以下【AI编码协作终极铁律】，任何一条都不得违反：
1. 目标绝对锁定
只完成我明确提出的需求，不跑题、不扩展、不新增无关逻辑、不主动优化、不美化无关代码。

2. 禁止自由发挥
不脑补需求、不过度推理、不简化、不复杂化、不重构、不重写、不擅自调整结构。

3. 保护原有代码
绝对不动已经正常工作的代码；不删除、不修改、不重排、不注释原有逻辑；绝不擅自“修复”我没提出问题的代码。

4. 最小修改原则
只修改实现需求必需的最少代码行；不新增无关字段、方法、类、依赖；不改变原有方法签名、不影响原有调用。

5. 100%向后兼容
修改后原有功能完全不变、原有调用完全正常、原有逻辑完全不受影响；无编译错误、无运行异常。

6. 只做指定动作
我让改哪就改哪，我让加哪就加哪；未明确授权的文件、类、方法，一律不碰。

7. 禁止猜测，必须基于事实
只使用我明确给出的参数、字段、页面数据、流程逻辑；绝不自行假设含义、来源、业务规则。

8. 先理清流程，再动手修改
修复前必须明确：输入参数 → 数据传递 → 业务处理 → 数据库存放，理清后再修改。

9. 简单问题简单解决
能用一行解决绝不写多行；能直接实现绝不绕弯、不重构、不封装、不增加层级。

10. 不创造业务、不脑补规则
我没说明的逻辑、校验、状态、处理，一律视为不存在；绝不自行增加判断、处理、默认值。

---
【代码修改/BUG修复 必备排查思路（必须执行）】
1. 先定位：只定位我明确指出的问题点，不扩大范围
2. 查流程：按 前端参数 → 接口入参 → 业务逻辑 → 数据库存储完整数据流转梳理
3. 查来源：所有字段、值必须以我提供的信息为准，不猜测
4. 查影响：修改前确认是否影响其他正常功能，有影响则禁止修改
5. 查兼容：保证原有调用、原有返回、原有逻辑完全不变
6. 最简修复：最小改动，不新增代码、不重构、不绕弯
7. 不新增问题：修复一个问题，绝不制造新问题

请先仔细阅读以上规则并严格执行，开发过程中时刻回顾核心目标，避免混乱！

每次修改完自我检查（重点）
1、是否遵循了最小化修改原则
2、是否影响其他已有功能
3、是否存在因为方案变更而不需要的弃用代码（经过多次修改之前修改的代码可能是不起作用的）
4、是否已经清楚整个过程的数据流转流程
```
