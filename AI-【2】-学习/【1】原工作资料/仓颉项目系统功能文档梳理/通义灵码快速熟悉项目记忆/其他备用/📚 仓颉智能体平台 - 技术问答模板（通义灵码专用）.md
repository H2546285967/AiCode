## 一、项目基础信息（固定前缀，必带）

```markdown
【项目名称】仓颉智能体平台
【核心定位】面向市场级应用的智能体工作流平台，支持多模型调用、RAG知识库检索、BPMN流程编排与SSE流式推送
【技术栈】Java 21 + Spring Boot 3.4.11 + PostgreSQL + Redis + MinIO + Camunda 工作流引擎
【核心模块】
1. 智能体配置：应用类型编码(0-5)、agent_app_base 配置表、is_current 版本管理
2. 工作流引擎：嵌入 agent-worker 服务，基于 BPMN 2.0 可视化建模，支持 SSE 流式推送
3. RAG 检索：文件上传 → 解析分块 → 向量化 + 关键词混合检索 → Rerank 重排序 → 召回 Top-K 片段
4. LLM 调用：ChatStrategySelector 策略选择器，适配 OpenAI、GLM、通义千问等多模型供应商
5. 发布权限：JWT 认证 + 组织/用户两级权限控制，市场级应用(app_type=5)自动全局共享
【编码规范】类名大驼峰、方法名小驼峰，异常统一抛出 BusinessException 并记录 ERROR 级日志
```

## 二、高频技术问答模板（可直接复用）

### 1. 工作流选型问答模板

**提问示例**：
> 基于以上项目信息，为什么我们选择 Camunda 作为工作流引擎，而不是其他轻量级框架？

**标准回答模板**：
> 我们选择 Camunda 是基于**技术栈集成度、业务控制力和企业级可靠性**的综合考量：
>
> 1. **深度集成**：原生 Java 内核，无缝嵌入 Spring Boot 3.4.11 的 agent-worker 服务，可直接管控流程变量、节点生命周期与事务，适配我们微服务架构的可观测性需求。
>
> 2. **可视化协作**：Camunda Modeler 提供专业 BPMN 图形编辑器，支持拖拽设计条件分支、并行网关等复杂流程，降低开发者与业务分析师的协作成本。
>
> 3. **企业级能力**：原生支持事务管理、任务调度、历史数据追溯与高可用部署，满足市场级应用对流程可靠性、可审计性的要求，这是轻量级框架（如 Flowable 简化版）无法覆盖的。
>
> 4. **扩展性**：支持自定义任务节点（如 SSE 推送、LLM 调用），可与我们的 RAG 检索、权限体系深度联动，符合智能体工作流的长期演进方向。
>
> 简言之，Camunda 是「可嵌入、可观测、可扩展」的企业级流程自动化平台，而非单纯的工作流库，完美匹配我们的产品定位。

### 2. RAG 检索设计问答模板

**提问示例**：
> 基于以上项目信息，我们的 RAG 检索为什么要采用「向量检索 + 关键词检索 + Rerank 重排序」的混合方案？

**标准回答模板**：
> 该设计是为了**弥补单一检索方式的缺陷，最大化提升问答准确率**：
>
> 1. **向量检索的局限**：擅长语义匹配，但对低频专业术语（如 `substring_index`、代码片段）召回率低，容易遗漏关键精确信息。
>
> 2. **关键词检索的补位**：通过倒排索引实现精确术语匹配，确保代码、专有名词等核心信息 100% 被召回，解决向量检索的盲区。
>
> 3. **Rerank 重排序的价值**：融合两种检索结果后，通过大模型对候选片段进行相关性打分，过滤噪声、优化顺序，最终只返回 Top-K 高质量片段给 LLM，避免简单拼接导致的结果混乱，保障最终回答的精准度。
>
> 这套「混合检索 + 重排序」是面向专业知识库场景的行业最佳实践，也是我们追求极致问答体验的核心设计。

### 3. LLM 调用策略问答模板

**提问示例**：
> 基于以上项目信息，ChatStrategySelector 策略选择器的设计目标是什么？它如何适配多模型供应商？

**标准回答模板**：
> ChatStrategySelector 的核心目标是**解耦 LLM 调用与业务逻辑，实现模型的灵活切换与灰度发布**：
>
> 1. **策略抽象**：定义统一的 LLM 调用接口，封装不同模型（OpenAI、GLM、通义千问等）的请求参数、流式响应格式与异常处理。
>
> 2. **动态路由**：根据应用配置（app_type、模型版本）、用户权限与流量策略，自动选择最优模型实例，支持按比例灰度切换新模型。
>
> 3. **容错降级**：当某模型服务不可用时，自动切换到备用模型，保障智能体服务的高可用性。
>
> 4. **日志埋点**：记录每次调用的模型类型、耗时与结果，为后续模型效果评估与迭代提供数据支撑。

### 4. 发布权限控制问答模板

**提问示例**：
> 基于以上项目信息，我们的发布权限控制是如何实现的？市场级应用(app_type=5)有什么特殊逻辑？

**标准回答模板**：
> 权限体系基于**JWT 认证 + 组织/用户两级权限**实现，核心逻辑如下：
>
> 1. **身份认证**：用户请求携带 JWT Token，系统解析出用户 ID、组织 ID 与角色信息。
>
> 2. **权限校验**：根据应用类型（app_type）与组织归属，校验用户是否具备查看/编辑/发布权限。
>
> 3. **市场级特殊逻辑**：当 app_type=5 时，应用自动标记为「市场级」，全局共享给所有组织与用户，无需单独授权，便于快速推广标准化智能体模板。
>
> 4. **操作审计**：所有发布/编辑操作均记录操作日志，包含操作者、操作时间与变更内容，满足合规审计要求。
