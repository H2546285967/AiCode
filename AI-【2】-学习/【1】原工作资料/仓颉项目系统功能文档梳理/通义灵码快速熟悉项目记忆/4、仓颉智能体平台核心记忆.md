## 一、智能体配置

### 🔍 检索标签
智能体配置、应用类型、配置表、版本管理、agent_app_base、agent_app_config

### 📌 核心信息

| 类型编码 | 名称     | 核心能力                | 配置存储表            |
| :------- | :------- | :---------------------- | :-------------------- |
| 0        | 知识问答 | RAG 检索+LLM 回答       | agent_app_config      |
| 1        | 工作流   | BPMN 流程编排           | agent_workflow_config |
| 3        | 对话流   | 多节点对话引导          | agent_workflow_config |
| 4        | 智能问数 | NL2SQL 数据库查询       | nl2sql_db_app         |
| 5        | 智能体   | MCP 工具调用/市场级应用 | agent_app_config      |

**配置结构：**

- `agent_app_base`：应用基本信息（名称、类型、状态、最新版本）
- `agent_app_config`：LLM 配置、Prompt、知识库绑定、文件上传设置
- `agent_app_version`：版本管理（预发布 v1.0 时间戳版，正式 v0.1/v0.2 递增版）
- `is_current` 字段标识当前使用版本

### 📝 高频场景

1. **新增应用**：先初始化 `agent_app_base` 基础信息 → 按类型写入对应配置表 → 生成 `agent_app_version` 版本记录
2. **版本切换**：修改 `agent_app_version` 中 `is_current` 字段为 1，同步更新 `agent_app_base` 最新版本

### ❓ 常见问题

| 问题                 | 排查思路                                                     |
| :------------------- | :----------------------------------------------------------- |
| 配置修改后不生效？   | 检查 `is_current` 字段是否指向最新版本；工作流类型需确认是否重新部署到 Worker |
| 智能问数配置找不到？ | 确认配置表为 `nl2sql_db_app`，而非 `agent_app_config`        |

## 二、技术栈

### 🔍 检索标签
技术栈、Java21、SpringBoot3.4.11、PostgreSQL、Redis、MinIO、Camunda、模块化架构

### 📌 核心信息

**核心技术栈：**

- **语言**：Java 21
- **框架**：Spring Boot 3.4.11
- **数据库**：PostgreSQL（业务数据）
- **缓存**：Redis（会话/缓存）
- **文件存储**：MinIO
- **工作流引擎**：Camunda（BPMN）
- **向量数据库**：算法服务提供的向量检索

**模块化架构（6 大模块群）：**

| 模块                             | 核心职责                           |
| :------------------------------- | :--------------------------------- |
| 系统服务 (agent-system)          | 用户、组织、权限、工作空间         |
| 智能体构建服务 (agent-builder)   | 应用管理、对话、知识库、模型、工具 |
| 工作流引擎 (agent-worker)        | BPMN 执行、节点调度                |
| 插件服务 (agent-plugin)          | 数据库、图表、消息、时间、公式插件 |
| RAG 适配器 (agent-rag-adapter)   | 向量检索适配                       |
| 模型适配器 (agent-model-adapter) | LLM 调用适配                       |

### 📝 高频场景

1. **模块依赖排查**：`agent-builder` 依赖 `agent-model-adapter` 实现 LLM 调用，依赖 `agent-rag-adapter` 实现向量检索
2. **环境搭建**：需提前部署 PostgreSQL（初始化业务表）、Redis（配置会话过期时间）、MinIO（创建文件存储桶）

### ❓ 常见问题

| 问题                       | 排查思路                                                     |
| :------------------------- | :----------------------------------------------------------- |
| Camunda 工作流节点不执行？ | 检查 `agent-worker` 模块是否启动；BPMN 文件是否符合 Camunda 语法规范 |
| MinIO 文件上传失败？       | 检查 `agent-builder` 模块的 MinIO 配置（endpoint/accessKey/secretKey）；存储桶权限是否为公共读 |

## 三、工作流

### 🔍 检索标签
工作流、节点类型、变量管理、流式响应、Camunda、部署Worker、WorkerStreamResponseV1

### 📌 核心信息

**节点类型：**

| 节点类型                | 用途                      |
| :---------------------- | :------------------------ |
| DirectReplyService      | 直接回复（静态/动态内容） |
| AgentLLMService         | 大模型节点                |
| ToolWorkflowService     | 工具/子流程调用           |
| FileAnalysisService     | 文件解析                  |
| LogicProcessService     | 条件分支                  |
| QuestionClassifyService | 问题分类器                |

**执行机制：**

```
开始节点 → 输入参数解析 → 节点执行 → 变量传递 → 输出结果
                ↓
        SSE 流式推送 (思考 + 回答)
```

**变量管理：**

- **全局变量**：`delegateExecution.setVariable(key, value)`
- **节点局部变量**：`WorkflowUtil.getCurrentNodeVariableName()` + 变量名

**发布流程：**

1. 保存配置到 `agent_workflow_config`（BPMN + config_v2）
2. 生成版本号到 `agent_app_version`
3. 工作流应用需部署到 Worker：`workflowClient.deployWorkflow()`
4. 更新应用状态为已发布

**流式响应数据结构：**

```java
WorkerStreamResponseV1 {
    messageId: "消息 ID"
    sessionId: "会话 ID"
    role: "assistant"
    think: "思考过程"
    context: "回答内容"
    totalTokens: 1500
    times: 2.5  // 思考时间 (秒)
}
```

### 📝 高频场景

1. **自定义工作流节点**：继承对应 Service 抽象类 → 实现 execute 方法 → 注册到 Camunda 节点工厂
2. **流式响应调试**：监听 SSE 连接，校验 think/context 字段是否按批次返回

### ❓ 常见问题

| 问题             | 排查思路                                                     |
| :--------------- | :----------------------------------------------------------- |
| 变量传递失败？   | 确认变量名是否统一（全局/局部区分）；检查 `delegateExecution` 是否为有效实例 |
| 工作流部署失败？ | 检查 BPMN 文件格式；确认 `agent-worker` 服务与 Camunda 引擎连通性 |

## 四、RAG 检索

### 🔍 检索标签
RAG、向量检索、文件解析、LargeParserHandler、RagProviderAdapter、KnowledgeConfig

### 📌 核心信息

**完整流程：**

```
用户上传文件 → 文件解析 → 分块处理 → 向量化 → 向量数据库
                                          ↓
用户提问 → 混合检索 (向量 + 关键词) → Rerank 重排序 → TopK → LLM 生成答案
```

**关键组件：**

| 组件                 | 职责                                               |
| :------------------- | :------------------------------------------------- |
| `RagProviderAdapter` | RAG 接口适配器（向量检索适配）                     |
| `LargeParserHandler` | 大文件异步解析处理器（MinerU/StructParser 解析器） |
| `FileInfoService`    | 文件管理服务（上传/解析状态跟踪）                  |

**检索配置参数：**

```java
KnowledgeConfig {
    knowledgeId: "知识库 ID"
    topK: 5                    // 返回数量
    score: 0.9                 // 阈值
    embedding_fn: {...}        // 嵌入模型配置
}
```

**文件解析策略：**

- **大文件**：异步解析队列 → MinerU/StructParser 解析器
- **小文件**：同步解析
- **支持格式**：pdf, docx, txt, md, excel 等

### 📝 高频场景

1. **大文件解析优化**：配置异步队列线程池大小；监控 `LargeParserHandler` 解析超时任务
2. **检索效果调优**：调整 topK（3-10）或 score 阈值（0.8-0.95）；优化混合检索权重

### ❓ 常见问题

| 问题                     | 排查思路                                                     |
| :----------------------- | :----------------------------------------------------------- |
| 文件解析后检索不到内容？ | 检查分块大小是否合理；确认向量化接口调用是否成功；查看 score 阈值是否过高 |
| Rerank 重排序无结果？    | 确认 RAG 适配器与算法服务的向量检索接口连通性；检查知识库 ID 是否绑定正确 |

## 五、LLM 调用

### 🔍 检索标签
LLM调用、多模型适配、ChatStrategySelector、OneApiClient、DashScopeClient、模型配置

### 📌 核心信息

**多模型适配策略：**

```
应用配置 → 提取模型信息 → {模型供应商}
                          ↓
  OpenAI/vLLM/GLM → OneApiClient (OpenAI 兼容接口)
  通义千问 → DashScopeClient (官方 SDK)
  Spring AI → SpringAiClient (Spring AI 抽象)
                          ↓
              ChatStrategySelector 策略选择器
                          ↓
              流式输出 Flux<String> / 普通响应
```

**支持的供应商：** OpenAI、GLM、vLLM、通义千问、Spring AI

**模型配置示例：**

```json
{
  "provider": "openai",
  "apiBase": "https://api.openai.com/v1",
  "apiKey": "sk-xxx",
  "model": "gpt-4",
  "completionParams": {
    "temperature": 0.6,
    "topP": 0.9,
    "maxTokens": 2000,
    "stream": true
  }
}
```

**关键实现：** `ChatStrategySelector`（模型调用策略选择器）

### 📝 高频场景

1. **新增模型适配**：实现 `SpringAiClient` 接口 → 注册到 `ChatStrategySelector` → 配置 provider 标识
2. **流式调用调试**：开启 `stream: true`，监听 `Flux<String>` 流式返回结果

### ❓ 常见问题

| 问题                     | 排查思路                                                     |
| :----------------------- | :----------------------------------------------------------- |
| 通义千问调用失败？       | 检查 `DashScopeClient` 的 apiKey 是否有效；确认模型名称（如 `qwen-turbo`）是否正确 |
| temperature 调整无效果？ | 确认 `completionParams` 已传递到 LLM 调用接口；部分模型需重启应用生效 |

## 六、发布权限

### 🔍 检索标签
发布权限、版本管理、JWT、agent_app_permission、多渠道发布、全局市场应用

### 📌 核心信息

**版本管理策略：**

- **预发布版本**：v1.0（时间戳版，用于测试）
- **正式发布版本**：v0.1/v0.2...（递增版，用于生产）
- `is_current` 字段标识当前使用版本

**权限控制体系：**

- **认证**：JWT Token
- **两级权限**：组织权限 + 用户权限
- **知识库权限**：私有/公开/临时三种模式
- **应用访问权限**：基于 `agent_app_permission` 表控制

**特殊场景（全局市场应用 app_type=5）：**

```java
if (MARKET_GLOBAL_CODE.equals(workspaceId)) {
    // 全局市场应用，所有人可读
    agentAppPermissionEntity = new AgentAppPermissionEntity()
        .setAppId(appId)
        .setTargetId("all")
        .setTargetType(ORG)
        .setAppPermission(AppBasePermissionEnum.READER.getCode());
    agentAppPermissionService.save(agentAppPermissionEntity);
}
```

**多渠道发布：**

- **Web 端**：免认证 URL / 用户校验 URL
- **钉钉**：机器人回调 → Markdown 卡片
- **飞书、微信公众号/企微**

### 📝 高频场景

1. **权限配置**：新增应用后，在 `agent_app_permission` 表写入组织/用户权限；全局应用需设置 `targetId="all"`
2. **多渠道适配**：钉钉机器人需配置回调地址，返回 Markdown 格式的响应内容

### ❓ 常见问题

| 问题                 | 排查思路                                                     |
| :------------------- | :----------------------------------------------------------- |
| 应用访问权限不足？   | 检查 `agent_app_permission` 表是否有对应用户/组织的权限记录；确认 JWT Token 是否过期 |
| 全局市场应用不可见？ | 确认 `app_type=5`；检查 `workspaceId` 是否为 `MARKET_GLOBAL_CODE`；验证权限记录是否写入 `all` 标识 |
