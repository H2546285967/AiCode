# 📚 **云库同步功能完整说明文档**

## 一、功能概述

### 1.1 功能定位
实现从**第三方云库**同步文件夹结构和文件到**本地知识库**,支持全量同步和增量同步。

### 1.2 核心价值
- ✅ 自动同步云库的文件夹结构和文件
- ✅ 支持增量同步，减少重复数据传输
- ✅ 自动检查并修复知识库创建者权限
- ✅ 详细的同步日志记录，便于问题排查
- ✅ 实时推送同步进度和结果

## 二、系统架构

### 2.1 技术栈
- **后端框架**: Spring Boot + MyBatis Plus
- **数据库**: PostgreSQL
- **对象存储**: MinIO (`http://10.16.2.97:9000/`)
- **缓存**: Redis (`10.16.0.166:30507`)
- **HTTP 客户端**: OkHttp

### 2.2 核心数据表

#### **1. file_other_open_data** - 云库文件映射表
```sql
-- 存储云库文件/文件夹的元数据映射
关键字段:
- meta_id: 云库文件/文件夹 ID
- file_name: 文件/文件夹名称
- file_id: 本地系统文件 ID
- knowledge_id: 知识库 ID
- created: 是否已创建 (0-未创建，1-已创建)
- state: 同步状态 (0-待同步，1-同步中，2-同步完成)
- only_read: 是否只读 (0-需要同步，1-只读不同步)
- code: 文件编码 (带层级关系)
- parent_meta_id: 父级云库 ID
- parent_meta_ids: 所有父级云库 ID 链
```


#### **2. file_extend** - 文件扩展配置表
```sql
-- 存储本地文件与云库的映射关系
关键字段:
- file_id: 本地文件 ID
- other_config: 云库映射配置 (格式：knowledgeId#metaId)
```


#### **3. file_info** - 文件信息表
```sql
-- 本地文件系统的基础信息
关键字段:
- id: 文件 ID
- file_name: 文件名称
- file_type: 文件类型 (3-文件夹，其他 - 文件)
- catalog_id: 所属目录 ID
- delete_flag: 删除标记
```


#### **4. file_metadata** - 文件元数据表
```sql
-- 文件的元数据信息
关键字段:
- file_id: 文件 ID
- delete_flag: 删除标记 (0-正常，1-已删除)
```


#### **5. file_other_open_data_detail** - 同步详情表
```sql
-- 记录每次同步的详细信息
关键字段:
- knowledge_id: 知识库 ID
- create_time: 同步时间 (用于增量同步的时间节点)
```


#### **6. base_log** - 操作日志表
```sql
-- 记录同步操作日志
关键字段:
- module_name: 模块名称 ("同步日志")
- ext_1: 扩展字段 1(存储 knowledgeId)
- content: 日志内容
- user_id: 操作用户 ID
```

## 三、核心流程

### 3.1 同步主流程 (init 方法)

```
1️⃣ 检查是否有进行中的同步任务
   ↓
2️⃣ 记录同步日志 (base_log)
   ↓
3️⃣ 异步执行同步任务 (fileSyncTaskExecutor)
   ↓
4️⃣ 模拟超级管理员登录
   ↓
5️⃣ 获取云库目录树 (getTree)
   ↓
6️⃣ 批量保存目录数据 (addBatch)
   ↓
7️⃣ 初始化文件夹结构 (initFolder)
   ↓
8️⃣ 检查并修复知识库创建者权限
   ↓
9️⃣ 标记需要同步的文件夹 (state=0)
   ↓
🔟 逐个文件夹同步文件 (sync)
   ↓
1️⃣1️⃣ 更新同步日志
   ↓
1️⃣2️⃣ 推送同步结果 (WebSocket)
```


### 3.2 文件夹初始化流程 (initFolder)

```
1️⃣ 查询待处理的文件夹 (foldersToProcess)
   ↓
2️⃣ 查询已存在的文件夹映射 (hasFolder)
   ↓
3️⃣ 过滤已删除的文件夹
   ↓
4️⃣ 创建 folderMap 和 hasFolderMap
   ↓
5️⃣ 遍历处理每个文件夹:
   ├─ 5.1 检查是否已存在但 fileId 为空
   ├─ 5.2 查找父级目录:
   │   ├─ 优先查 hasFolderMap (已存在的映射)
   │   ├─ 其次查 folderMap (本次待创建的)
   │   └─ 都找不到则报错
   ├─ 5.3 创建文件夹 (addFolder)
   ├─ 5.4 更新 file_other_open_data (fileId, created)
   └─ 5.5 处理异常情况 (文件夹已存在)
   ↓
6️⃣ 为已存在的文件夹补充权限
```


### 3.3 文件同步流程 (sync 方法)

```
1️⃣ 调用云库 API 获取文件列表
   ↓
2️⃣ 遍历文件列表:
   ├─ 检查文件是否需要同步
   ├─ 下载文件内容
   ├─ 上传到 MinIO
   ├─ 创建本地文件记录
   └─ 更新映射关系
   ↓
3️⃣ 统计同步结果 (新增/更新/删除)
```

## 六、数据映射关系

### 6.1 云库 → 本地系统

```
云库目录树
  ├─ metaId → file_other_open_data.meta_id
  ├─ fileName → file_other_open_data.file_name
  ├─ parentMetaId → file_other_open_data.parent_meta_id
  ├─ parentMetaIds → file_other_open_data.parent_meta_ids
  └─ permissions → file_permission (权限表)

本地文件夹创建后:
  ├─ file_info.id → file_other_open_data.file_id
  ├─ file_info.id → file_extend.file_id
  └─ knowledgeId#metaId → file_extend.other_config
```


### 6.2 文件夹层级关系

```
知识库 (knowledgeId)
  └─ 一级目录 (level=1)
      └─ 二级目录 (level=2)
          └─ 三级目录 (level=3)
              └─ 文件
```

## 七、异常处理

### 7.1 常见异常及处理

#### **1. BusinessException("上级目录不存在")**
- **原因**: 父级目录在 `file_extend` 表中没有映射记录
- **解决**: 
  - 清理 `file_other_open_data` 表的脏数据
  - 确保父级目录先创建

#### **2. BusinessException("已存在同步任务")**
- **原因**: 同一知识库有 state=2 的同步任务进行中
- **解决**: 等待任务完成或手动清理状态

#### **3. FOLDER_EXIST_ERROR**
- **原因**: 文件夹已存在
- **处理**: 自动捕获并查找已存在的文件夹，更新映射关系

### 7.2 脏数据清理 SQL

```sql
-- 1. 检查 file_other_open_data 中的脏数据
SELECT 
    foor.meta_id,
    foor.file_name,
    foor.file_id,
    foor.created
FROM file_other_open_data foor
LEFT JOIN file_info fi ON foor.file_id = fi.id
WHERE foor.knowledge_id = 'xxx'
  AND foor.created = 1
  AND fi.id IS NULL;

-- 2. 清理脏数据
UPDATE file_other_open_data
SET created = 0,
    file_id = NULL,
    update_time = NOW()
WHERE knowledge_id = 'xxx'
  AND created = 1
  AND file_id NOT IN (SELECT id FROM file_info);

-- 3. 清理 file_extend 孤儿数据
DELETE FROM file_extend
WHERE file_id NOT IN (SELECT id FROM file_info);
```

## 九、权限管理

### 9.1 知识库创建者权限检查

```java
// 每次同步自动检查并修复
checkAndFixKnowledgeOwnerPermission(knowledgeId, knowledgeCreateUserId)

检查逻辑:
1. 查询知识库创建者是否已有 owner 权限
2. 如果没有，自动添加:
   - owner 权限 (255)
   - manager 权限 (从 owner 派生)
   - read 权限 (从 owner 派生)

### 9.2 文件夹权限设置

权限来源:
- 云库 API 返回的 permissions 数组
- 自动派生 (owner → manager → read)
```

## 十、同步策略

### 10.1 全量同步 vs 增量同步

#### **全量同步**
- 首次同步或没有同步历史记录
- 同步所有云库文件和文件夹

#### **增量同步**
- 基于 `file_other_open_data_detail` 表的最新同步时间
- 只同步 `updateTime` 之后的变更
- 通过云库 API 的 `updateTime` 参数控制

### 10.2 同步状态管理

```
state 字段含义:
- 0: 待同步 (初始状态或需要重新同步)
- 1: 同步中 (正在处理)
- 2: 同步完成

only_read 字段含义:
- 0: 需要同步 (读写模式)
- 1: 只读 (不同步)
```

## 十三、常见问题 FAQ

### Q1: 同步失败提示"上级目录不存在"怎么办？
**A**: 
1. 检查 `file_other_open_data` 表是否有脏数据
2. 检查 `file_extend` 表是否有映射记录
3. 清理脏数据后重启服务

### Q2: 为什么同步日志显示成功但看不到文件？
**A**:
1. 检查 `file_metadata` 表的 `delete_flag` 是否为 0
2. 检查 `file_extend` 表的映射关系是否正确
3. 确认 MinIO 中文件是否存在

### Q3: 如何触发增量同步？
**A**:
- 系统自动基于上次同步时间判断
- 修改 `file_other_open_data_detail` 表的 `create_time`
- 云库 API 会根据 `updateTime` 参数返回增量数据

### Q4: 权限检查失败怎么办？
**A**:
1. 查看日志 `[权限检查]` 相关输出
2. 确认知识库创建者 ID 是否正确
3. 手动执行 `checkAndFixKnowledgeOwnerPermission` 方法
