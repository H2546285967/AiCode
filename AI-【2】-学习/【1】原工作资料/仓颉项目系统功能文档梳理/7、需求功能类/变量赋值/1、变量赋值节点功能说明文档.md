# 变量赋值节点功能说明文档

## 一、需求背景

### 1.1 业务场景

在仓颉智能体平台的工作流/对话流编排中，存在以下变量管理需求：

1. **变量初始化**：在流程开始时设置初始值（如计数器清零、状态标志设置）
2. **变量重置**：在流程执行过程中重新设置变量值
3. **变量引用传递**：将一个节点的输出赋值给全局变量，供后续多个节点使用
4. **类型转换**：将字符串转换为数字、布尔值等特定类型
5. **对话状态管理**：在对话流中维护和更新对话状态变量

### 1.2 现有问题

在变量赋值节点开发之前，用户面临以下痛点：

- ❌ **无法在流程中动态设置变量**：只能依赖开始节点的输入参数
- ❌ **变量作用域受限**：节点输出只能通过 `${id_xxxGROUPvar}` 引用，不够灵活
- ❌ **缺少类型转换能力**：字符串和数字之间的转换需要额外节点处理
- ❌ **批量赋值繁琐**：设置多个变量需要多个节点或复杂配置

### 1.3 需求目标

提供一个专门的**变量赋值节点**，实现：

- ✅ **批量赋值**：一次配置多个变量的赋值操作
- ✅ **灵活引用**：支持常量、变量引用、混合表达式
- ✅ **类型安全**：支持自动类型转换和类型检查
- ✅ **全局作用域**：赋值的变量可被后续所有节点访问
- ✅ **对话流支持**：在对话流中持久化变量，跨多轮对话使用

## 二、功能设计

### 2.1 核心功能

#### 2.1.1 批量变量赋值

支持一次性配置多个变量的赋值操作。

**示例配置**：
```json
{
  "assignments": [
    {
      "variableName": "counter",
      "variableValue": "0",
      "variableType": "number"
    },
    {
      "variableName": "user_name",
      "variableValue": "${id_inputGROUPuser_input}",
      "variableType": "string"
    },
    {
      "variableName": "is_validated",
      "variableValue": "true",
      "variableType": "boolean"
    }
  ]
}
```

#### 2.1.2 变量值来源

支持三种变量值来源：

1. **常量值**：直接指定的固定值
   - 字符串：`"hello"`, `"张三"`
   - 数字：`123`, `3.14`
   - 布尔值：`true`, `false`

2. **变量引用**：引用其他节点的输出
   - 全局变量：`${varName}`
   - 节点变量：`${id_xxxGROUPvarName}`
   - 系统变量：`${sys_query}`, `${sys_auth}`

3. **混合表达式**：常量 + 变量引用
   - `"用户输入：${input}"`
   - `"第${counter}次尝试"`

#### 2.1.3 类型转换

支持显式指定目标类型，自动进行类型转换。

**支持的类型**：
| 类型名称 | Java 类型 | 示例值 |
|---------|----------|--------|
| `string` | `String` | `"hello"` |
| `number` | `Double` | `123.45` |
| `integer` | `Integer` | `123` |
| `boolean` | `Boolean` | `true` |
| `object` | `Map` | `{"key": "value"}` |
| `array` | `List` | `[1, 2, 3]` |
| `auto` | 自动识别 | 根据值自动判断 |

#### 2.1.4 变量作用域

赋值的变量具有**双重作用域**：

1. **全局变量**：`varName`
   - 所有后续节点可访问
   - 在对话流中可跨多轮对话保持

2. **节点变量**：`id_currentGROUPvarName`
   - 当前节点可访问
   - 用于节点输出引用

### 2.2 技术架构

#### 2.2.1 类结构

```
VariableAssignmentService
├── 继承：AgentBaseAbstractDelegate
├── 注解：@Component("variableAssignmentService")
├── 依赖：
│   ├── WorkflowUtil（变量解析和设置）
│   ├── WorkflowTypeMappingUtil（类型映射）
│   ├── TypeConvertUtils（类型转换）
│   └── JsonUtil（JSON 解析）
└── 核心方法：doExecute(DelegateExecution)
```

#### 2.2.2 执行流程

```
开始
  ↓
步骤 1: 获取节点配置（getExtProperties）
  ↓
步骤 2: 解析赋值配置列表（JSON 解析）
  ↓
步骤 3: 遍历赋值项
  ├─ 提取配置（变量名、值、类型）
  ├─ 验证变量名有效性
  ├─ 步骤 4: 解析变量值（处理变量引用）
  ├─ 步骤 5: 类型转换（如果指定类型）
  ├─ 步骤 6: 设置全局变量（setDefaultAndNodeVariables）
  └─ 步骤 7: 记录赋值日志
  ↓
结束
```

#### 2.2.3 与其他节点的关系

```
开始节点 → [变量赋值节点] → 大模型节点 → 输出节点
              ↓
         设置全局变量
              ↓
         后续节点都可通过 ${varName} 引用
```

## 三、实现方案

### 3.1 代码实现

#### 3.1.1 核心代码结构

```java
@Component(value = "variableAssignmentService")
public class VariableAssignmentService extends AgentBaseAbstractDelegate {
    
    // 配置常量
    public static final String ASSIGNMENTS = "assignments";
    public static final String VARIABLE_NAME = "variableName";
    public static final String VARIABLE_VALUE = "variableValue";
    public static final String VARIABLE_TYPE = "variableType";
    
    @Override
    protected void doExecute(DelegateExecution delegateExecution) throws Exception {
        // 步骤 1: 获取节点配置
        Map<String, Object> extProperties = getExtProperties(delegateExecution);
        Object assignmentsObj = extProperties.get(ASSIGNMENTS);
        
        // 步骤 2: 解析赋值配置列表
        List<Map<String, Object>> assignments = parseAssignments(assignmentsObj);
        
        // 步骤 3-7: 遍历执行赋值
        for (Map<String, Object> assignment : assignments) {
            // 3.1-3.2: 提取配置并验证
            String varName = (String) assignment.get(VARIABLE_NAME);
            Object varValue = assignment.get(VARIABLE_VALUE);
            String varType = (String) assignment.get(VARIABLE_TYPE);
            
            // 步骤 4: 解析变量值（处理变量引用）
            Object realValue = parseVariableValue(delegateExecution, varValue);
            
            // 步骤 5: 类型转换
            if (指定了类型) {
                realValue = convertToType(realValue, varType);
            }
            
            // 步骤 6: 设置全局变量
            WorkflowUtil.setDefaultAndNodeVariables(delegateExecution, varName, realValue);
            
            // 步骤 7: 记录日志
            log.info("变量赋值完成：{} = {}", varName, realValue);
        }
    }
}
```

#### 3.1.2 关键方法说明

**1. 获取节点配置**
```java
Map<String, Object> extProperties = getExtProperties(delegateExecution);
```
- **来源**：继承自 `AgentBaseAbstractDelegate`
- **用途**：从 BPMN 的 `camunda:properties` 中获取配置
- **复用**：文本提取节点第 88、93 行

**2. 解析变量引用**
```java
Object realValue = WorkflowUtil.replaceAllVariable2NewContent(delegateExecution, valueStr);
```
- **工具类**：`WorkflowUtil`
- **功能**：将 `${varName}` 替换为实际值
- **复用**：文本提取节点第 84 行

**3. 类型转换**
```java
Class<?> javaType = WorkflowTypeMappingUtil.getJavaType(varType);
realValue = TypeConvertUtils.convert(realValue, javaType);
```
- **类型映射**：`WorkflowTypeMappingUtil.getJavaType()`
- **类型转换**：`TypeConvertUtils.convert()`
- **复用**：VariableExtractService 第 166 行

**4. 设置变量**
```java
WorkflowUtil.setDefaultAndNodeVariables(delegateExecution, varName, realValue);
```
- **功能**：同时设置全局变量和节点变量
- **复用**：文本提取节点第 245、267 行

## 四、技术细节

### 4.1 变量解析机制

#### 4.1.1 变量引用格式

| 格式 | 说明 | 示例 |
|------|------|------|
| `${varName}` | 全局变量引用 | `${counter}` |
| `${id_xxxGROUPvar}` | 节点变量引用 | `${id_startGROUPsys_query}` |
| `${sys_xxx}` | 系统变量引用 | `${sys_auth}` |

#### 4.1.2 解析流程

```
输入："用户输入：${input}"
  ↓
WorkflowUtil.replaceAllVariable2NewContent()
  ↓
1. 正则匹配：\$\{([^}]+)\}
2. 提取变量名：input
3. 获取变量值：delegateExecution.getVariable("input")
4. 替换： "用户输入：张三"
  ↓
输出："用户输入：张三"
```

---

### 4.2 类型转换机制

#### 4.2.1 转换规则

**String → Number**：
```java
"123" → 123.0 (Double)
"3.14" → 3.14 (Double)
"abc" → 转换失败，抛出异常
```

**Number → String**：
```java
123 → "123"
3.14 → "3.14"
```

**String → Boolean**：
```java
"true" → true
"false" → false
其他值 → 转换失败
```

**Auto 模式**：
```java
"123" → Integer (如果是整数格式)
"3.14" → Double (如果是小数格式)
"true" → Boolean
"abc" → String (默认)
```

#### 4.2.2 类型检查

```java
// 检查是否可以转换
if (WorkflowTypeMappingUtil.canConvertToJavaType(value, targetType)) {
    // 执行转换
    realValue = TypeConvertUtils.convert(realValue, javaType);
} else {
    // 转换失败处理
    throw new CommonException(400, "类型转换失败");
}
```

---

### 4.3 错误处理

#### 4.3.1 异常类型

| 异常场景 | 异常信息 | 处理方式 |
|---------|---------|---------|
| 配置格式错误 | "赋值配置格式错误" | 抛出 CommonException(400) |
| 变量名为空 | "跳过无效赋值配置" | 跳过该项，继续处理 |
| 类型转换失败 | "变量类型转换失败" | 抛出 CommonException(400) |
| 变量引用不存在 | 返回 null | 设置为 null，记录警告日志 |

#### 4.3.2 错误日志

```java
try {
    // 类型转换
    realValue = TypeConvertUtils.convert(realValue, javaType);
} catch (Exception e) {
    log.error("变量 {} 类型转换失败：{}", varName, realValue, e);
    throw new CommonException(400, "变量类型转换失败：" + e.getMessage());
}
```

## 八、常见问题

### 8.1 FAQ

#### Q1: 变量命名有什么规范？

**A**: 变量名应遵循以下规则：
- 只能包含字母、数字、下划线
- 不能以数字开头
- 不能使用保留关键字（如 `sys_query`, `id_xxxGROUP` 等）
- 建议使用小驼峰命名（如 `userName`, `totalCount`）

---

#### Q2: 如何引用其他节点的输出？

**A**: 使用 `${id_节点 IDGROUP 变量名}` 格式：
```
节点 A 的输出变量：output
在变量赋值节点中引用：${id_nodeAGROUPoutput}
```

---

#### Q3: 类型转换失败会怎样？

**A**: 类型转换失败会抛出异常，流程会进入异常处理分支：
```
变量赋值节点 → 转换失败 → 异常捕获 → 推送错误消息 → 流程终止
```

**解决方案**：
- 使用 `auto` 类型，让系统自动识别
- 在赋值前添加校验节点，确保数据类型正确
- 在配置界面添加类型提示

---

#### Q4: 对话流中变量会持久化吗？

**A**: 当前实现中，变量只在**当前流程实例**内有效。如果需要在多轮对话中保持变量，需要：

**方案 1**：在每轮对话开始时重新赋值
```
对话 1: 变量赋值 (step=1) → 处理 → 结束
对话 2: 变量赋值 (step=2) → 处理 → 结束
```

**方案 2**：扩展 Redis 持久化（待开发）
```java
// 在变量赋值时持久化到 Redis
if ("chatflow".equals(type)) {
    redisTemplate.opsForHash().put("session:" + sessionId, varName, realValue);
}
```

---

#### Q5: 可以赋值 null 值吗？

**A**: 支持赋值 null 值：
```json
{
  "variableName": "optional_field",
  "variableValue": null,
  "variableType": "auto"
}
```

**效果**：
- 变量会被设置为 null
- 后续节点访问时返回 null
- 不会抛出异常

---

### 8.2 故障排查

#### 问题 1：变量赋值后无法访问

**排查步骤**：
1. 检查变量名是否正确
2. 检查赋值节点是否在访问节点之前执行
3. 查看日志确认赋值是否成功
4. 验证变量作用域（全局 vs 节点局部）

**解决方案**：
- 确保使用 `${varName}` 引用全局变量
- 检查 BPMN 流程顺序
- 查看日志：`grep "变量赋值完成" logs/agent-worker.log`

---

#### 问题 2：类型转换失败

**排查步骤**：
1. 检查源数据类型
2. 检查目标类型是否支持
3. 查看转换失败的错误信息
4. 验证数据格式

**解决方案**：
- 使用 `auto` 类型自动识别
- 在配置界面添加数据类型校验
- 添加数据预处理节点

---

#### 问题 3：变量引用解析失败

**排查步骤**：
1. 检查变量引用格式是否正确（`${varName}`）
2. 检查被引用的变量是否存在
3. 检查引用顺序（不能引用后续节点的变量）
4. 查看日志确认解析结果

**解决方案**：
- 使用正确的引用格式
- 确保被引用变量已设置
- 调整节点执行顺序
