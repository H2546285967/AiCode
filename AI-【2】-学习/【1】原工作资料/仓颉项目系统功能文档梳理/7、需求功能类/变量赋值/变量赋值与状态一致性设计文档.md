# 变量赋值与对话内全局变量状态一致性机制设计文档

## 1. 背景与问题

在仓颉智能体平台的对话流（Chatflow）中，用户期望通过“变量赋值节点”修改的全局变量（如 `userName`, `phone` 等）能够在其后的所有对话和节点中持续生效。然而，在之前的实现中存在一个关键缺陷：

- **问题描述**：当用户在一个会话中通过变量赋值节点修改了全局变量后，再次输入信息时，系统打印或读取到的仍是修改前的旧值。
- **根本原因**：该问题是由于变量的修改仅停留在流程执行的内存上下文中，而没有将变更持久化到数据库的会话表（`agent_chat_session`）中。当下一次请求到来时，系统会从数据库重新加载变量，导致覆盖了之前在内存中的修改，从而出现状态不一致。

## 2. 设计目标

确保在单个对话流中，全局变量的状态能够保持一致性和连续性。具体目标如下：

1.  **跨执行持久化**：任何通过变量赋值节点对全局变量的修改，都必须被保存到数据库，以确保其在多次流程执行间依然有效。
2.  **启动时加载**：每次对话流启动时，都必须优先从数据库加载最新的会话变量，作为本次执行的初始状态。
3.  **节点独立性**：修复方案不应影响其他节点的正常运行，遵循系统的模块化架构约束。
4.  **数据一致性**：保证内存中的变量状态与数据库中的存储状态最终一致，避免因事务回滚等问题导致数据丢失。

## 3. 解决方案与核心改动

为解决上述问题，我们对 `VariableAssignmentService` 和 `WorkflowServiceImpl` 两个核心服务类进行了协同修改。

### 3.1 核心组件：`VariableAssignmentService`

此服务是变量赋值节点的具体实现，负责处理赋值逻辑。其关键改动在于引入了 `updateSessionVariable` 方法，以确保变量变更能持久化到数据库。

#### 关键代码段 (VariableAssignmentService.java)

```java
/**
 * 更新会话的 variable 字段到数据库
 * 确保变量在多次执行之间保持
 */
@Transactional(propagation = Propagation.REQUIRES_NEW)
protected void updateSessionVariable(DelegateExecution delegateExecution, String varName, Object varValue) {    
    try {
        // 1. 获取会话 ID
        String sessionId = delegateExecution.getVariable("sys_sessionId") == null ? ""
                : delegateExecution.getVariable("sys_sessionId").toString();

        if (sessionId.isEmpty()) {
            log.warn("无法更新会话变量：sessionId 为空");
            return;
        }

        // 2. 从数据库查询当前会话实体
        AgentChatSessionMapper sessionMapper = SpringContextUtil.getBean(AgentChatSessionMapper.class);
        AgentChatSessionEntity sessionEntity = sessionMapper.selectById(sessionId);

        if (sessionEntity == null) {
            log.warn("无法更新会话变量：会话不存在，sessionId={}", sessionId);
            return;
        }

        // 3. 解析现有的 JSON 格式变量字段
        Map<String, Object> variableMap = new HashMap<>(); // 初始化空Map
        String existingVar = sessionEntity.getVariable();
        if (existingVar != null && !existingVar.isEmpty()) {
            try {
                Map<String, Object> parsedMap = JsonUtil.getJsonToBean(existingVar, Map.class);
                if (parsedMap != null) {
                    variableMap.putAll(parsedMap); // 合并现有数据
                }
            } catch (Exception e) {
                log.warn("解析会话变量 JSON 失败，使用空对象。原始内容: {}", existingVar, e);
            }
        }

        // 4. 将新值写入Map，并序列化回JSON字符串
        variableMap.put(varName, varValue != null ? varValue.toString() : "");
        sessionEntity.setVariable(JsonUtil.getObjectToString(variableMap));

        // 5. 将更新后的实体保存回数据库
        sessionMapper.updateById(sessionEntity);

        log.info("会话变量更新成功：sessionId={}, {}={}", sessionId, varName, varValue);
    } catch (Exception e) {
        log.error("更新会话变量失败：{}", varName, e);
        // 不抛出异常，避免影响主流程执行
    }
}
```

#### 执行流程

1.  **获取上下文**：从流程执行上下文 (`delegateExecution`) 中获取 `sys_sessionId`。
2.  **查询实体**：使用 `AgentChatSessionMapper` 通过 `sessionId` 查询数据库中的会话记录。
3.  **合并数据**：将数据库中已有的 `variable` 字段（JSON格式）反序列化为 `Map`，然后将新的变量名/值对存入该Map。
4.  **持久化**：将更新后的Map重新序列化为JSON字符串，并调用 `updateById` 方法将其保存回数据库。

> **关键设计点**：该方法使用 `@Transactional(propagation = Propagation.REQUIRES_NEW)` 注解，创建了一个独立的新事务。这是为了避免如果主流程后续发生回滚，不会连带使这次至关重要的变量更新也一并被撤销，从而保证了数据的可靠性。

### 3.2 启动时加载：`WorkflowServiceImpl`

为了实现“启动时加载”的目标，我们在 `WorkflowServiceImpl.startChatflow` 方法中添加了从数据库加载变量的逻辑。这确保了每次对话开始时，都能拿到最新、最全的变量快照。

#### 关键代码段 (WorkflowServiceImpl.java)

```java
// 【对话流专用】从数据库加载会话变量，确保变量状态一致性
if (sessionId != null && !sessionId.isEmpty()) {
    startParams.put("sys_sessionId", sessionId);

    try {
        AgentChatSessionEntity sessionEntity = agentChatSessionMapper.selectById(sessionId);
        if (sessionEntity != null && sessionEntity.getVariable() != null && !sessionEntity.getVariable().isEmpty()) {
            Map<String, Object> sessionVars = JsonUtil.getJsonToBean(sessionEntity.getVariable(), Map.class);
            if (sessionVars != null) {
                for (Map.Entry<String, Object> entry : sessionVars.entrySet()) {
                    // 直接变量名（兼容已有逻辑）
                    startParams.put(entry.getKey(), entry.getValue());

                    // 带 globalGROUP 前缀的全局变量（供回复节点使用）
                    String globalVarName = "globalGROUP" + entry.getKey();
                    startParams.put(globalVarName, entry.getValue());
                }
            }
        }
    } catch (Exception e) {
        log.warn("从数据库加载会话变量失败：{}", sessionId, e);
    }
}
```

该逻辑在 `startChatflow` 的早期阶段执行，它会将数据库中的变量注入到流程的启动参数 (`startParams`) 中，成为整个流程执行的初始环境。

## 4. 整体工作流

综合以上改动，完整的变量流转和状态保持工作流如下：

1.  **初始化**：用户首次发起对话，`startChatflow` 被调用，由于是新会话，数据库中无历史变量，流程以默认值启动。
2.  **修改**：流程执行至“变量赋值节点”，`VariableAssignmentService.doExecute` 被触发。在设置完内存中的变量后，会立即调用 `updateSessionVariable` 将变更**持久化到数据库**。
3.  **下一次交互**：当用户再次输入信息，触发一个新的流程执行时，`startChatflow` 会首先根据 `sessionId` **从数据库加载上一次保存的所有变量**，并注入到新的流程实例中。
4.  **读取**：后续的任何节点（如大模型节点、回复节点）在读取变量时，都将基于这个包含了最新值的初始状态进行，因此可以正确地读取到修改后的值。

## 5. 验证与结论

通过上述设计，我们成功解决了全局变量状态不一致的问题。该方案具有以下优点：

- **高可靠性**：利用数据库作为单一可信来源，彻底解决了内存状态易失的问题。
- **强一致性**：实现了“修改 -> 持久化 -> 加载 -> 读取”的闭环，保证了数据的一致性。
- **低侵入性**：改动集中在两个核心服务，未对其他节点造成负面影响，符合系统架构原则。

此设计已成为仓颉智能体平台处理对话内全局变量的标准模式。
