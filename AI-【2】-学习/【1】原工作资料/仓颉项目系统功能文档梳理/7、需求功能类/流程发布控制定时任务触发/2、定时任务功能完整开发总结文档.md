# 📋 定时任务功能完整开发总结文档

**涉及模块**：agent-worker（Camunda BPMN 引擎）、agent-build-core（运营日志展示） 
**核心功能**：工作流定时任务的创建、执行、监控、取消发布删除  

---

## 🎯 一、需求背景与目标

### 1.1 业务需求

**核心目标**：
1. ✅ 实现工作流的定时触发功能（Timer Start Event）
2. ✅ 工作流监测日志正常展示定时执行日志
3. ✅ 点击定时任务日志能查看详细执行流程
4. ✅ 取消发布时彻底删除定时任务，避免误执行

**应用场景**：
- 📊 每天早上 8 点自动统计昨日数据
- 🔄 每小时自动执行数据同步
- 📈 每周一生成周报
- 🗂️ 定期清理过期数据

---

### 1.2 技术挑战

**主要难点**：
1. ⚠️ Camunda Timer Start Event 不会经过普通 startEvent，系统变量缺失
2. ⚠️ 定时任务没有 SSE 连接，推送逻辑需要特殊处理
3. ⚠️ 定时任务没有用户 ID 和 sessionId，数据关联困难
4. ⚠️ 取消发布时只暂停流程定义，定时任务仍会执行（严重 Bug）

### 2.1 定时任务完整生命周期（6 个阶段）

```
┌─────────────────────────────────────────────────────────────┐
│                    定时任务完整生命周期                      │
└─────────────────────────────────────────────────────────────┘

1️⃣ 发布应用（创建定时任务）
   ↓
   前端配置 cron 表达式（如：0 29 15 * * ?）
   ↓
   BPMN 解析 → 提取 TimerStartEvent 节点
   ↓
   Camunda 创建定时作业
   ↓
   ACT_RU_JOB 表插入记录
   - ID_: 805785124306040965
   - TYPE_: 'timer'
   - PROCESS_DEF_ID_: 'process_805760277769239685:3:805785124280875141'
   - DUEDATE_: '2026-03-23 15:29:00.0'
   - SUSPENSION_STATE_: 1 (激活)

2️⃣ 定时任务触发（自动执行）
   ↓
   Camunda 作业执行器扫描 ACT_RU_JOB
   ↓
   匹配 DUEDATE_ <= 当前时间 的作业
   ↓
   StartNodeListener 拦截
   ↓
   【关键】初始化系统变量（sys_appId, sys_workflowId 等）
   ↓
   【关键】创建助手消息记录（用于保存执行结果）
   ↓
   执行节点链（LLM/代码/赋值等）
   ↓
   创建流程实例
   ↓
   ACT_HI_PROCINST 插入记录
   - ID_: 805784588152356357
   - PROC_DEF_KEY_: 'process_805760277769239685'
   - START_TIME_: '2026-03-23 15:20:18.569'
   - END_TIME_: '2026-03-23 15:20:18.613'
   - DURATION_: 44 (毫秒)

3️⃣ 记录执行日志
   ↓
   EndNodeListener 拦截
   ↓
   【关键】跳过 SSE 推送（定时任务无连接）
   ↓
   【关键】保存节点状态到数据库
   ↓
   更新助手消息记录（content 字段包含所有变量）
   ↓
   ACT_HI_JOB_LOG 插入记录
   - ID_: 805784588152356358
   - PROCESS_DEF_KEY_: 'process_805760277769239685'
   - JOB_STATE_: 'SUCCESS'
   - REMOVAL_TIME_: '2026-03-23 15:20:18.613'

4️⃣ 查询监测日志
   ↓
   前端调用 /api/operation/workflow/operateLog
   ↓
   AgentOperationServiceImpl 查询
   ↓
   【关键】识别定时任务（bizId=null && createUserId=null）
   ↓
   【关键】设置渠道="定时任务"，bizId=procInstId
   ↓
   返回列表数据

5️⃣ 查看执行详情
   ↓
   前端点击日志行
   ↓
   调用 /api/workflow/detail/{procInstId}
   ↓
   WorkflowServiceImpl.getProcessTrack()
   ↓
   【关键】过滤 outputs，只保留 exception 字段
   ↓
   返回节点追踪详情

6️⃣ 取消发布（删除定时任务）
   ↓
   前端点击"取消发布"
   ↓
   调用 suspendTimerTaskByProcDefId()
   ↓
   【关键修复】遍历删除所有 ACT_RU_JOB 记录
   ↓
   【关键修复】暂停流程定义
   ↓
   ACT_RU_JOB 记录被删除（0 行）
   ↓
   ACT_RE_PROCDEF.SUSPENSION_STATE_ = 2 (暂停)
```

### 2.3 定时任务 vs 手动触发详细对比表

| 对比维度           | 定时任务（Timer Start Event）           | 手动触发（Start Event）                    | 说明                             |
| ------------------ | --------------------------------------- | ------------------------------------------ | -------------------------------- |
| **触发方式**       | Camunda 自动扫描 ACT_RU_JOB 表          | 前端点击试运行按钮                         | 定时任务无需人工干预             |
| **启动节点**       | timerStartEvent                         | startEvent                                 | BPMN 节点类型不同                |
| **businessKey**    | null                                    | workflowId（有值）                         | ⚠️ 定时任务的 businessKey 为 null |
| **createUserId**   | null                                    | 实际用户 ID（如：admin）                   | ⚠️ 定时任务没有创建人             |
| **sys_sessionId**  | ""（空字符串）                          | 实际 sessionId                             | 定时任务没有会话概念             |
| **sys_appId**      | 从 processDefinitionKey 解析            | 从前端参数获取                             | 都需要设置                       |
| **sys_workflowId** | businessKey（null）或 processInstanceId | 前端传入的 workflowId                      | 定时任务需要用 processInstanceId |
| **SSE 连接**       | 无                                      | 有                                         | ⚠️ 定时任务不能推送 SSE           |
| **消息记录创建**   | StartNodeListener 中创建                | WorkflowServiceImpl.startChatflow() 中创建 | 都需要创建                       |
| **渠道标识**       | "定时任务"                              | "应用编排"                                 | 前端显示不同                     |
| **推送逻辑**       | 跳过 SSE，只保存数据库                  | 正常 SSE 推送                              | 定时任务跳过推送                 |
| **运行日志用户列** | "-"（无用户）                           | 实际用户名（如：管理员）                   | 前端展示不同                     |

### 3.1 问题 1：定时任务触发时流程变量缺失

#### 📋 问题描述

**现象**：
- 定时任务触发时，`sys_appId`、`sys_workflowId` 等系统变量为 null
- 后续节点（LLM、代码节点等）无法获取必要参数
- 流程执行失败或结果异常

#### 🔍 根因分析

**数据流分析**：
```
普通手动触发流程：
前端 → WorkflowServiceImpl.startChatflow() 
     → 设置 sys_appId, sys_workflowId 等变量
     → runtimeService.startProcessInstanceByKey()
     → StartNodeListener.doExecute()
     → 节点已经是 startEvent，跳过变量初始化
     → 执行后续节点

定时任务触发流程：
Camunda Job Executor 
     → startProcessInstanceByMessage()
     → StartNodeListener.doExecute()
     → ⚠️ 节点是 timerStartEvent，不是 startEvent
     → ⚠️ 原有代码只处理 startEvent，跳过变量初始化
     → ❌ 后续节点获取不到系统变量
```


**关键代码（修复前）**：
```java
// StartNodeListener.java - 修复前
String nodeType = delegateExecution.getBpmnModelElementInstance().getElementType().getTypeName();

if ("startEvent".equals(nodeType)) {
    // 只初始化 startEvent 的变量
    Map<String, Object> startVariables = getStartNodeOutputParams(delegateExecution);
    delegateExecution.setVariables(startVariables);
}
// ❌ TimerStartEvent 没有处理
```


#### ✅ 解决方案（完整代码）

**修复代码**：
```java
// StartNodeListener.java - 修复后
@Override
protected void doExecute(DelegateExecution delegateExecution) {
    String nodeType = delegateExecution.getBpmnModelElementInstance().getElementType().getTypeName();
    String processDefinitionKey = delegateExecution.getProcessDefinitionKey();
    String processInstanceId = delegateExecution.getProcessInstanceId();
    String businessKey = delegateExecution.getBusinessKey();
    
    log.info("===== StartNodeListener 执行开始 =====");
    log.info("节点类型：{}", nodeType);
    log.info("流程实例 ID: {}", processInstanceId);
    log.info("businessKey: {}", businessKey);
    
    // 处理普通 Start Event 的情况（手动触发）
    if ("startEvent".equals(nodeType)) {
        log.info("普通 Start Event 触发，初始化系统变量");
        Map<String, Object> startVariables = getStartNodeOutputParams(delegateExecution);
        delegateExecution.setVariables(startVariables);
    }
    
    // 【关键修复】处理 Timer Start Event 的情况（定时任务）
    if ("timerStartEvent".equals(nodeType)) {
        log.info("Timer Start Event 触发，初始化系统变量");
        
        // 1. 从 processDefinitionKey 提取 appId
        String appId = extractAppIdFromProcessDefKey(processDefinitionKey);
        delegateExecution.setVariable("sys_appId", appId);
        
        // 2. 设置 workflowId（businessKey 为 null 时使用 processInstanceId）
        String workflowId = businessKey != null ? businessKey : processInstanceId;
        delegateExecution.setVariable("sys_workflowId", workflowId);
        
        // 3. 定时任务没有会话概念，sessionId 为空
        delegateExecution.setVariable("sys_sessionId", "");
        
        // 4. 使用 processInstanceId 作为 instanceId
        delegateExecution.setVariable("sys_instanceId", processInstanceId);
        
        // 5. 设置其他系统变量
        delegateExecution.setVariable("sys_type", "workflow");
        delegateExecution.setVariable("sys_userId", "");
        delegateExecution.setVariable("sys_createTime", new Date());
        
        log.info("定时任务系统变量初始化完成：appId={}, workflowId={}", appId, workflowId);
    }
}

/**
 * 从流程定义 Key 中提取应用 ID
 * 格式：process_{appId}
 */
private String extractAppIdFromProcessDefKey(String processDefinitionKey) {
    if (processDefinitionKey != null && processDefinitionKey.startsWith("process_")) {
        return processDefinitionKey.substring("process_".length());
    }
    return "";
}
```

```

    // 【关键修复】3. 暂停流程定义
    repositoryService.suspendProcessDefinitionById(procDefId);
    log.info("流程定义已暂停：{}", procDefId);

## 🎓 六、关键知识点总结

### 6.1 Camunda Timer Start Event 机制

**核心要点**：
1. ⚠️ Timer Start Event 不会经过普通的 startEvent
2. ⚠️ 需要手动初始化系统变量（sys_appId, sys_workflowId 等）
3. ⚠️ businessKey 为 null，需要使用 processInstanceId
4. ⚠️ 没有 sessionId 概念（空字符串）
5. ⚠️ 没有 SSE 连接（跳过推送）

---

### 6.2 三个关键 ID 对比

| ID 类型               | 手动触发                 | 定时任务                 | 用途                     |
| --------------------- | ------------------------ | ------------------------ | ------------------------ |
| **businessKey**       | workflowId（有值）       | null                     | 关联消息记录的 parent_id |
| **processInstanceId** | 始终有值（Camunda 生成） | 始终有值（Camunda 生成） | 流程实例唯一标识         |
| **bizId**             | 使用 businessKey         | 使用 processInstanceId   | 前端查询详情的参数       |

## ⚠️ 七、常见错误与避免

### 7.1 错误 1：只暂停不删除定时任务

```java
// ❌ 错误：只暂停，定时任务仍在
managementService.suspendJobByProcessDefinitionKey(procDefId);
```


**后果**：定时任务会继续执行

**修复**：遍历删除所有定时任务

---

### 7.2 错误 2：使用错误的查询条件

```java
// ❌ 错误：使用 processDefinitionKey（会匹配到所有版本）
managementService.createJobQuery()
    .processDefinitionKey(procDefKey)
    .list();
```


**后果**：可能删除其他版本的定时任务

**修复**：使用 `processDefinitionId` 精确匹配

```java
// ✅ 正确：使用 processDefinitionId
managementService.createJobQuery()
    .processDefinitionId(procDefId)
    .list();
```

### 7.3 错误 3：未处理空列表

// ❌ 错误：未判空

// ✅ 正确：添加判空
if (CollectionUtils.isEmpty(jobs)) {

##  九、遗留问题与注意事项

### 9.1 部分定时任务显示异常

**现象**：少数定时任务显示 "推送节点结束状态失败"

**可能原因**：
- EndNodeListener 的 push 方法在特定情况下抛出异常
- 数据库保存失败

**建议**：
- 添加更详细的异常日志
- 考虑在定时任务中完全跳过 SSE 推送

---

### 9.2 资源竞争问题

**注意**：
- ⚠️ 定时任务执行间隔不能太短（建议≥5 秒）
- ⚠️ 避免多个定时任务同时执行导致数据库锁竞争
- ⚠️ 监控数据库连接池使用情况

---

### 9.3 环境隔离

**配置**：
- ✅ 本地环境：允许运行定时任务

    job-execution:
      enabled: true

  bpm:
    job-execution:
      enabled: false  # 线上禁用
```
