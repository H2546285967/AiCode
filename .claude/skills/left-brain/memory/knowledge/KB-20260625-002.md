# KB-20260625-002

---
name: KB-20260625-002
description: M9 任务复杂度评分（dispatcher 0-10 数字打分）
category: feature_full
---

**目标**：dispatcher.js 加 `scoreComplexity(task)`，让派/不派有量化依据，不只靠关键词硬规则。

**API**：
```js
const { scoreComplexity } = require('./dispatcher');
const r = scoreComplexity('全面排查多模块 BUG');
// r = { score: 6.8, band: 'gray_zone', breakdown: { fileCount, moduleCount, dispatchKeyword, dontKeyword, taskType } }
```

**三档阈值**（`RULES.scoring`）：
- `score ≤ 3` → `no_dispatch`
- `4 ≤ score ≤ 7` → `gray_zone`
- `score ≥ 8` → `dispatch`

**加性打分**（不是 max，保留多信号叠加）：
| 信号 | 权重 |
|:-----|:-----|
| 文件数 | × 0.6 |
| 模块数 | × 1.2 |
| should_dispatch 关键词 | +3 |
| dont_dispatch 关键词 | -1.5 |
| 强任务类型（bug_fix / refactor / migration / feature_full / multi_module） | +2 |
| 单人处理类型（explanation / question / single_edit） | -1 |
| 钳制 | Math.max(0, Math.min(10, round(score, 1))) |

**集成**：`decide()` 所有 return 路径加 `complexity_score` + `complexity_band` 字段，**不破坏现有 17/17 集成测试 + 65/65 单元测试**。

**测试**：`test-dispatcher-scoring.js` 43/43 通过（10 维度：返回结构 / 分数范围 / 三档阈值 / 灰区 / 边界钳制 / 关键词影响 / 任务类型影响 / decide 集成 / 分数一致性 / CLI 输出）。

**预期业务效果**：小任务不派率从"硬规则"提升到"硬规则 + 评分"双保险，下游灰区决策可读 `complexity_score` 选派 1 / 2 / 3 Agent。

**未做**（留 M10+）：把 score 接入 `suggested_action.agents`（现在是固定 2，应改 Math.ceil(score / 3)）。
