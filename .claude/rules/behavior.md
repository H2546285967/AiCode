## 📖 文件读取策略

- **按需读取** — 用户提到哪个文件就读哪个，不提前遍历整个目录
- **批量读取前确认** — 需要读多个文件时，先列清单让用户确认再逐个读取
- **优先搜索定位** — 使用 Grep/Glob 搜索而非盲目遍历目录
- **长篇只读关键部分** — 大文件只读取需要的行号范围
- **全局扫描前必须询问** — 以下操作属于全局扫描，执行前必须先向用户说明原因并请求授权：
  - 用 `find`、`glob **`、`ls -R` 等遍历整个仓库
  - 用 `Grep` 跨文件搜索未限定路径或模式的字符串（如"找所有 TODO"）
  - 用 `Glob` 匹配大量文件（如 `**/*.js`）
  - 批量读取用户未明确指定的文件
  - 用户请求中出现"所有"、"整个仓库"、"全部"、"每个文件"等范围词时，先确认范围再执行

## ✍️ 输出控制

- **回答精炼** — 避免冗长的解释性输出
- **代码直接可用** — 不输出大段注释的"教学式"代码
- **默认最简方案** — 用户没特别要求时，给最简洁的方案

## ✂️ 简洁优先（来自 Karpathy）

> **用最少的代码解决问题。不要过度推测。**

对抗过度工程的显式纪律：

- 不要添加要求之外的功能
- 不要为一次性代码创建抽象
- 不要添加未要求的"灵活性"或"可配置性"
- 不要为不可能发生的场景做错误处理
- 如果 200 行代码可以写成 50 行，重写它
- 新增代码前先问："这是用户要求的，还是我想加的？"

**检验标准：** 资深工程师会觉得这过于复杂吗？如果是，简化。

## 🎯 精准修改（来自 Karpathy）

> **只碰必须碰的。只清理自己造成的混乱。每一行修改都必须追溯到用户请求。**

编辑现有代码时：

- 不要"改进"相邻的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你更倾向于不同的写法
- 如果注意到无关的死代码，提一下 —— 不要删除它
- 当你的改动产生孤儿代码时，删除因你的改动而变得无用的导入/变量/函数
- 不要删除预先存在的死代码，除非被要求

**检验标准：** 如果这一行改动与用户请求无关，就不应该出现在 diff 里。

- 全新任务时建议 `/clear` 清空旧上下文
- **new task = new session**，除非强关联

## 🤖 子代理自主调度

当任务符合以下**任意一条**时，自动拆分子代理并行处理，用户无需手动指定：

| 触发条件 | 说明 |
|:---------|:------|
| 涉及 5+ 个文件的新建/修改 | 拆出编码子代理，主 context 只做协调 |
| 前后端 + 数据库全栈 | planner 规划 → 后端/前端子代理并行编码 |
| 包含"调研/研究/比较"等探索性任务 | explorer 单独调研，主 context 继续工作 |
| 用户说"复杂"、"全面"、"完整"等词 | 默认启用子代理模式 |

### 调度原则

- **planner** → 架构设计、技术选型、方案对比
- **explorer** → 代码搜索、文档调研、目录结构理解
- **reviewer** → 代码审查、Bug 发现、安全检查
- **并行**：不互相依赖的任务同时跑，互不阻塞
- **结论返回**：子代理只返回结果，中间过程不污染主 context

## 🌿 Git 分支管理

**默认在 `main` 分支工作**（个人工程无需多分支）。详见 [git-branch.md](git-branch.md)：
- ✅ 在 main 上 commit 自由操作
- ⚠️ 切换 / 创建 / 删除分支必须先问用户
- ❌ 禁止未经同意开新分支
