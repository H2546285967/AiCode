## 🎯 为什么需要这条规则

**AI 默认做类比推理**（找训练数据里类似的方案微调）—— 快但跳过了"这个问题真的应该这么解吗？"。

**两个 Prompt 的本质**：
1. **第一性原理**：强制打断类比推理，逼 AI 回到问题本质重新推导
2. **对抗式审查**：开 N 个 Agent 站在恶意用户/黑客角度找漏洞

**这两个加起来 = 生成 + 验证的完整闭环**，是 Vibe Coding 的两大基石。

## 🧠 编码前思考（Think Before Coding · 来自 Karpathy）

### 一句话

> **不要假设。不要隐藏困惑。呈现权衡。适时提出异议。**

### 为什么需要

LLM 编码时最危险的倾向是**默默选择一种解释然后执行**：用户说"优化一下"，AI 就按自己的理解重写；遇到歧义，不澄清直接猜。Karpathy 把这叫"代你做错误假设"。

这个原则强制把隐性推理显性化：

- **明确说明假设** — 如果不确定，询问而不是猜测
- **呈现多种解释** — 当存在歧义时，列出至少两种理解让用户选
- **适时提出异议** — 如果用户方案明显更复杂，说出更简单的方法
- **困惑时停下来** — 指出不清楚的地方并要求澄清，而不是硬编

### 与第一性原理的关系

**编码前思考是 0.5a 的前置动作**。先停止默认类比推理（编码前思考），再追问本质（第一性原理）。

| 顺序 | 动作 | 问题 |
|:-----|:-----|:-----|
| 0.5a-1 | 编码前思考 | "我理解对了吗？有没有更简单的路？" |
| 0.5a-2 | 第一性原理 | "这个问题的本质是什么？" |

## 🟢 第一性原理（First Principles · 生成前必问）

### 一句话

> **在 prompt 末尾加一句："从第一性原理出发"**

### 用法

```
[你的任务描述]

—— 从第一性原理出发
```

### 触发场景

| 场景 | 是否必问 | 例 |
|:-----|:------:|:---|
| 🔴 设计架构 | ✅ 必须 | "从第一性原理出发，给这 3 个模块设计边界" |
| 🔴 找 BUG 根因 | ✅ 必须 | "不要治表，从第一性原理找真正的原因" |
| 🟡 修明显 bug | 🟡 建议 | 表层修复够用就不必 |
| 🟢 typo/小改 | ❌ 跳过 | 不必浪费 token |

### 4 类反模式（必避）

| ❌ 反模式 | ✅ 第一性原理做法 |
|:---------|:-----------------|
| "用 Redis 缓存"（行业共识）| "这个数据真的需要缓存吗？访问频率多少？" |
| "写个 React 组件"（训练数据类似）| "这个 UI 交互的本质是什么？状态模型对吗？" |
| "模仿 GitHub trending" | "我们的 L4 学习闭环和 GitHub 有什么本质差异？" |
| "加个中间层" | "这个中间层真的隔离了问题，还是把问题放大？" |

### 与现有机制的关系

| 已有 | 关系 |
|:-----|:-----|
| CLAUDE.md 最高指令「先问这能帮 Claude 变智能吗？」| ✅ 第一性原理的子集 |
| M48 neat-freak「毕业机制」| ✅ 已用第一性原理推导 |
| M49 deep-research「借鉴方法论核心，砍掉 PDF」| ✅ 已用第一性原理推导 |
| KB-001 Top 10「不要整体 clone」| ✅ 已用第一性原理推导 |
| ECC 评估「主体不借鉴 5 理由」| ✅ 已用第一性原理推导 |

**结论**：编码前思考 + 第一性原理在 AiCode 已经**半显式存在**，但**未作为统一方法论沉淀**。M52 把它升级为**显式方法论 + self-discipline 0.5a 必跑动作**；2026-07-01 进一步把 Karpathy「编码前思考」拆为 0.5a-1 前置动作。

## 🔴 对抗式审查（Adversarial Review · 完成前必跑）

### 一句话

> **开 N 个 Agent 站在"我要搞崩你这个系统"的角度并发审查**

### 用法（Claude Code）

```
开启多 Agent 对 [刚完成的功能 / 整个项目] 进行对抗式审查：
- 假设你是恶意用户 / 黑客 / 运维新手
- 找能搞崩系统的所有角度（输入异常 / 边界条件 / 并发 / 资源耗尽 / 时间污染）
- 找出后立即修复
```

### 触发场景

| 场景 | 是否必跑 | Agent 数建议 |
|:-----|:------:|:----------:|
| 🏁 里程碑（v1.X 完成）| ✅ **必须** | 4-8 |
| 🔴 大级（新增架构 / 跨模块）| ✅ **必须** | 2-4 |
| 🟡 小级（bug fix）| 🟡 建议 | 1-2 |
| 🟢 微小（typo）| ❌ 跳过 | — |

### 审查角度清单（每次必查 5 类）

| # | 角度 | 例 |
|:--|:-----|:---|
| 1 | **输入异常** | 50MB HTML / 负数 / Unicode 乱码 / 未来时间戳 |
| 2 | **边界条件** | 空数组 / 0 / NaN / null / undefined |
| 3 | **并发 / 资源** | 重试死循环 / OOM / 文件句柄泄漏 |
| 4 | **时间 / 状态污染** | 未来时间戳排第一 / 缓存穿透假阳性 |
| 5 | **部署 / 回滚** | 元数据丢失 / DB 不一致 / 路径回滚边界 |

### 与现有机制的关系

| 已有 | 关系 |
|:-----|:-----|
| `qa-reviewer` 子代理 | ✅ 对抗式审查的轻量子集 |
| `code-review` / `secondary-review` 命令 | ✅ 对抗式审查的入口 |
| `swarm-coordinator`（多 Agent 异构视角）| ✅ 对抗式审查的引擎（M31）|
| `/go` 第 3 阶段（review）| ✅ 流水线集成 |

**结论**：对抗式审查在 AiCode 已经**有工具无强制**。M52 把它升级为**self-discipline 0.5c 必跑动作**（🔴 大 / 🏁）。

## 📋 实战示例（来自 AIHOT 作者）

### 案例 1：第一性原理找根因（替代治表）

```
❌ 治表：
"OpenAI 抓取断了 → 单独修复 OpenAI 抓取 → 修好了"

✅ 第一性原理：
"为什么 OpenAI 抓取断了？
→ 因为底层流量路由机制有隐患（今年 4 月写的代码）
→ 未来任何信源都可能再断
→ 花了半天重构路由层 → 治本"
```

### 案例 2：对抗式审查找 BUG（防漏）

```
❌ 正常开发思维：
"写完 worker → 跑通测试 → 上线"
→ OOM 死循环没发现
→ 50MB HTML 输入把 worker 打爆 → 自动重试 → 又爆 → 无限循环

✅ 对抗式审查：
"假设你是恶意用户提交 50MB HTML
→ 走完整路径：入口 → worker → OOM → 重试 → 死循环
→ 找出根因 → 加大小限制 + 重试退避
→ 后续真碰到 100MB HTML → 安全"
```

### 案例 3：未来时间污染（AI 自己想不到的 BUG）

```
"如果文章发布时间是未来（比如明天），会发生什么？
→ 时间戳最新 → 排第一 → 推送用户 → 污染信息流
→ 这种 BUG 自己写代码根本不会想到
→ 对抗式审查才能问到"
```
