# AI Coding 开发范式完整指南

> 五大时代对比 + SDD 规范驱动开发实战手册

---

## 第一部分：AI Coding 五大时代对比

### 核心总览

| 时代 | 名称 | 核心模式 | 你的角色 | 主流工具 |
|:----:|:-----|:---------|:---------|:---------|
| **0** | 古法编程 | 全程手写代码 | 打字员、调试工 | VS Code / IDEA |
| **1** | 网页对话 | 复制 AI 代码片段粘贴到 IDE | 代码搬运工 | ChatGPT 网页版 |
| **2** | IDE Agent | AI 读本地仓库，长提示词全自动 | 需求提出者 | Cursor / Claude Code / MiniMax |
| **3** | SDD 规范驱动 | 独立 spec.md 为唯一真值，AI 按规范开发 | 系统意图定义者 | spec 文档 + Agent |
| **5** | Nimi Coding 契约治理 | 三层文件体系 + 机器强制校验 + 多 Agent 分层 | 契约制定者 | nimi-coding CLI |

### 递进关系

```
0. 自己手写代码
   ↓
1. 和 AI 聊天拿代码片段
   ↓
2. AI 直接操作本地完整项目（长提示词全自动）
   ↓
3. 独立文档固化需求（SDD）← 我们当前在这里
   ↓
5. 机器强制契约校验 + 分层多 Agent 治理（工业化）
```

### 各时代深度拆解

#### 时代 0：古法编程
- 流程：接到需求 → 手写代码 → 搜资料 → 手动调试
- 优点：可控性 100%
- 缺点：速度极慢，大量重复劳动

#### 时代 1：网页对话 AI
- 流程：网页提问 → 复制代码片段 → 粘贴到 IDE → 报错再问
- 优点：入门零成本
- 缺点：上下文割裂、AI 不懂你的项目

#### 时代 2：IDE Agent（当前主流）
- 流程：粘贴完整长提示词 → AI 自动读项目、写代码、自测、打包
- 优点：全自动生成，一套提示词复制即用
- **硬伤**：Review 疲劳 + 意图丢失（对话多了 AI 遗忘业务规则）

#### 时代 3：SDD 规范驱动（我们采用）
- 流程：需求 → Spec（意图）→ Plan（规划）→ Tasks（拆分）→ Code（代码）
- **核心理念**：代码是编译产物，Spec 才是源代码
- 优点：需求永久固化、不会丢失意图、复用 Spec 模板批量开发
- **硬伤**：Spec 是纯文字，无机器强制校验，仍需人工审核

#### 时代 5：Nimi Coding 契约治理
- 三层体系：`contracts/`（机器可校验契约）+ `spec/`（意图文档）+ `methodology/`（标准流程）
- 解决 SDD 硬伤：多 Agent 分层隔离（规划/编码/校验分离），杜绝伪成功
- 适合：工业化批量产出、高阶付费教程

### 选择建议

| 场景 | 推荐时代 |
|:-----|:---------|
| 偶尔做工具、拍短视频演示 | 时代 2（长提示词） |
| **稳定批量开发、长期迭代** | **时代 3（SDD）** ← 当前 |
| 工业化批量产出、追求极低审核成本 | 时代 5（Nimi Coding） |

## 第二部分：SDD 规范驱动开发实战手册

### SDD 核心理念

> **代码是事实（结果），Spec 是意图（源头）。Spec 才是源代码，代码只是编译产物。**

### SDD 强制工作流程（四阶段）

```
┌─────────────────────────────────────────────────────┐
│  阶段 1：方案输出                                      │
│  读取 Spec → 输出完整开发方案（结构/模块/流程/数据）    │
│  → 等待用户回复【确认方案】                             │
├─────────────────────────────────────────────────────┤
│  阶段 2：垂直切片编码（分层式已淘汰）                    │
│  不再按"DB→API→UI"分大阶段，改为按功能垂直切片          │
│  每个切片横跨 DB+API+UI，完成后端到端可验证              │
│  例：用户登录功能 = 用户表 + 登录接口 + 登录页面          │
├─────────────────────────────────────────────────────┤
│  阶段 3：自动自测                                      │
│  全部代码完成 → 模拟运行 → 排查 bug/越界/卡顿           │
│  → 自动修复所有问题                                    │
├─────────────────────────────────────────────────────┤
│  阶段 4：完整交付                                      │
│  输出 readme + 打包说明 + 分发方法                      │
│  → 交付可离线运行成品                                  │
└─────────────────────────────────────────────────────┘
```

### SDD Agent 系统提示词模板

```text
# 角色：SDD 规范驱动全栈开发 Agent

## 核心工作铁律（严格按顺序，禁止跳过）
1. 第一阶段：读取项目根目录 spec.md，以文档内容为唯一真值，禁止自由发挥增删功能。
2. 需求确认：输出完整开发方案（项目结构、模块拆分、逻辑流程、数据结构），等【确认方案】。
3. 编码开发：按功能垂直切片（每个切片横跨 DB+API+UI），分文件输出完整代码，标注路径，带详细中文注释，关键参数抽离配置区。
4. 自测修复：代码完成后模拟运行，自查 bug/边界/异常，自动修复。
5. 交付收尾：输出 readme + 打包说明，告知启动步骤。

## 强制约束
1. 关键参数统一写在代码顶部 CONFIG 配置区
2. 完善边界校验、动画过渡、点击防抖
3. 输出格式：
   【当前阶段】：xxx
   【阶段产出】：xxx
   【下一步】：等待确认/输出代码

## 禁止行为
1. 不输出方案直接写代码
2. 超出 Spec 规范擅自增加功能
3. 省略注释、省略运行说明
```

### SDD 输出格式规范

每个阶段必须使用统一格式：

```
【当前阶段】：Step 1 — 完整开发方案输出
【阶段产出】：项目结构、模块拆分、流程图、数据结构
【下一步】：等待用户回复【确认方案】
```

```
【当前阶段】：Step 2 — 分模块输出代码
【阶段产出】：index.html（完整代码，带 CONFIG + 中文注释）
【下一步】：等待用户确认 / 继续输出下一个文件
```

### SDD 使用流程

| 步骤 | 操作 |
|:-----|:-----|
| 1 | 新建项目文件夹，创建 `spec.md`，写入完整规范 |
| 2 | 发送 SDD Agent 系统提示词 |
| 3 | 发送执行指令："读取 spec.md，按 SDD 四阶段流程开发" |
| 4 | Agent 输出方案 → 你回复【确认方案】 |
| 5 | Agent 分模块输出代码 → 自测修复 |
| 6 | Agent 交付 readme + 打包说明 → 完成 |

### 两种模式

| 模式 | 适用场景 | 说明 |
|:-----|:---------|:-----|
| **SDD 标准方案** | 长期迭代、多次修改 | 独立 `spec.md` + 简短系统提示词 |
| **长提示词模式** | 一次性项目 | 把 spec 合并进提示词一次性发送 |

## 第三部分：在本工作空间中使用 SDD

### 何时使用 SDD

| 场景 | 是否用 SDD |
|:-----|:----------|
| 开发小游戏、工具类项目 | ✅ 使用，写 spec.md |
| 开发业务系统、完整应用 | ✅ 使用，写 spec.md |
| 简单脚本、一次性修改 | ❌ 不用，直接开发 |
| 修 Bug、代码优化 | ❌ 不用，直接对话 |

### SDD 项目模板

```
AI-【3】-项目开发/<项目名>/
├── spec.md           ← 需求规范（SDD 唯一真值）
├── CLAUDE.md         ← 项目指令（引用 AI-ClaudeCode-最佳实践精简.md）
├── index.html / src/ ← 代码（按 spec 开发）
├── readme.md         ← 运行说明
└── dist说明.txt      ← 打包分发
```
